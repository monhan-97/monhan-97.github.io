React18

如何升级 React18​

详情请参考官方升级指南

安装​

安装最的 React 版本

npm install react react-dom

或者你使用 yarn 来进行安装

yarn add react react-dom

Updates to Client Rendering APIs​

当你第一次安装 React 18 时，你会在控制台中看到一个警告：

http://localhost:3000
  • ReactDOM.render is no longer supported in React 18. Use createRoot instead. Until you switch to the new API, your app will behave as if it’s running React17. Learn more: https://reactjs.org/link/switch-to-createroot

React 18 启用了 concurrent渲染器 ，可以选择并发功能

// Before
import { render } from "react-dom";
const container = document.getElementById("app");
render(<App tab="home" />, container);

// After
import { createRoot } from "react-dom/client";
const container = document.getElementById("app");
const root = createRoot(container); // createRoot(container!) if you use TypeScript
root.render(<App tab="home" />);

React 18 还会将 unmountComponentAtNode 更改为 root.unmount

// Before
unmountComponentAtNode(container);

// After
root.unmount();

React 18 还从 render 中删除了回调，因为在使用 Suspense 时它通常没有预期的结果：

// Before
const container = document.getElementById("app");
render(<App tab="home" />, container, () => {
  console.log("rendered");
});

// After
function AppWithCallbackAfterRender() {
  useEffect(() => {
    console.log("rendered");
  });

  return <App tab="home" />;
}

const container = document.getElementById("app");
const root = createRoot(container);
root.render(<AppWithCallbackAfterRender />);

最后，如果您的应用程序使用 hydration 进行服务器端渲染 需要 hydrate 升级为 hydrateRoot

// Before
import { hydrate } from "react-dom";
const container = document.getElementById("app");
hydrate(<App tab="home" />, container);

// After
import { hydrateRoot } from "react-dom/client";
const container = document.getElementById("app");
const root = hydrateRoot(container, <App tab="home" />);

Updates to Server Rendering APIs​

在 React 18 中 react-dom/server API 需要完全支持 Suspense 的服务端渲染 和 Streaming SSR。

作为这些更改的一部分，React 18 弃用了旧的 Node streaming API, 因为它不支持新增的 Suspense 的服务端渲染

使用此 API 现在会发出警告

  • renderToNodeStream : Deprecated ⛔️️

代替的是, 对于 Node 环境中的流式传输 请使用:

  • renderToPipeableStream : New ✨

React 18 引入了一个新的 API，以支持带有 Suspense 的流式 SSR

  • renderToReadableStream : New✨

以下 API 将继续工作，但对 Suspense 的支持有限

  • renderToString: Limited ⚠️

  • renderToStaticMarkup:Limited ⚠️

Updates to TypeScript definitions​

如果您的项目使用 TypeScript,则需要将@types/react和 @types/react-dom 依赖项更新到最新版本。新类型更安全，并且可以捕获过去被类型检查器忽略的问题。 最显着的变化是 children prop 现在需要在定义 props 时明确列出，例如：

interface MyButtonProps {
  color: string;
  children?: React.ReactNode;
}

自动批处理​

什么是批处理​

React 18 将多个状态更新分组到一次渲染中以获得更好的性能

在 React 18 之前， React 只在 合成事件里面可以批量更新。 Promise、setTimeout 、原生事件处理（native event handlers） 或任何其他事件内部的更新默认情况下不会在 React 中进行批处理：

// 在 React 18 之前，只有 React 合成事件会被批处理

function handleClick() {
  setCount((c) => c + 1);
  setFlag((f) => !f);
  // React 只会在最后重新渲染一次
}

setTimeout(() => {
  setCount((c) => c + 1);
  setFlag((f) => !f);
  // React 将会 渲染两次,每个状态都需要更新一次
}, 1000);

<button onClick={handleClick}></button>;

从带有 createRoot 的 React 18 开始，所有更新都将自动批处理，无论它们来自何处。也就是说 Promise、setTimeout 、原生事件处理（native event handlers） 或者其他的任何事件 将和 React 事件内部更新相同的方式进行批处理

// React 18 更新后  `Promise`、`setTimeout` 、`Promise`、`setTimeout` 、`原生事件处理（native event handlers）`
// 处理函数的状态更新都会被合并成一次渲染

function handleClick() {
  setCount((c) => c + 1);
  setFlag((f) => !f);
  // React 最后只会渲染一次
}

setTimeout(() => {
  setCount((c) => c + 1);
  setFlag((f) => !f);
  // React 最后只会渲染一次
}, 1000);

<button onClick={handleClick}></button>;

如果不想批处理怎么办​

通常，批处理是安全的 可以提高我们应用程序的性能，但某些代码可能依赖于在状态更改后立即从 DOM 中读取某些内容。 对于这些用例，我们可以使用 ReactDOM.flushSync() 选择退出批处理：

import { flushSync } from "react-dom";

function handleClick() {
  flushSync(() => {
    setCounter((c) => c + 1);
  });
  // React 现在已经更新了 DOM
  flushSync(() => {
    setFlag((f) => !f);
  });
  // React 现在已经更新了 DOM
}

为 React 第三方库新增的 APIs​

useSyncExternalStore​

对于 React 的状态管理库 来说，它的首要问题不是如何改变状态，而是如何触发更新 不管是 Mobx 的 observable，还是 Redux 的单向数据流，它们本身并不会绑定使用的框架或场景， 因此不能直接用在 React 或其他 MVVM 框架上 于是有了 mobx-react 和 react-redux 将数据管理方案与 react 相结合

useSyncExternalStore 是一个 React Hook,可让我们订阅外部 store。

const snapshot = useSyncExternalStore(
  subscribe,
  getSnapshot,
  getServerSnapshot
);

在组件的顶层调用 useSyncExternalStore 以从外部 store 中读取值

返回 store 中数据的快照。需要传递两个函数作为参数：

  • subscribe: 需要订阅 store 并返回一个取消订阅的函数.
  • getSnapshot: 返回 store 中读取数据的快照。
  • 可选的 getServerSnapshot： 返回 store 中数据的初始快照的函数。它将会在服务器渲染期间和客户端上服务器渲染内容的 hydration(混合) 期间使用 服务器快照在客户端和服务器之间必须是相同的 否则在服务端渲染组件将引发错误

useTransition​

渲染的过程中 有些 UI 需要尽快更新(比如:input 输入框,下拉选择框) 而其他的 UI 渲染可以有较低的优先级(比如:过滤列表)

从 React 18 开始 你可以开启 concurrent 模式 - 允许将 UI 标记为高低优先级 或者可中断的低优先级

默认情况下 React 中的所有更新都是紧急的 当快速更新因大量更新而减慢的时候 可能会产生问题

我们可以使用 useTransition 这个 hook 将 UI 更新为低优先级 对于大量非紧急的更新特别有用

参数​

const [isPending, startTransition] = useTransition();
  • isPending:transition 是否在 pending
  • startTransition(callback): 允许在回调中把 UI 的更新状态标记为 transition

使用 useTransition 的时候 需要确保启用并发模式

import { useTransition } from "react";

function MyComponent() {
  const [isPending, startTransition] = useTransition();
  // ...

  const someEventHandler = (event) => {
    startTransition(() => {
      // updates as transitions
      setValue(event.target.value);
    });
  };

  return <HeavyComponent value={value} />;
}

Heavy UI updates as urgent​

紧急的大量 UI 更新 是比较影响用户体验的

在快速输入查询的时候 我们可以注意到打字滞后 并且 UI 在明显的时间段内 感觉没有响应

在用户输入的时候更新是一项快速执行的紧急任务 但是通过突出匹配项来更新列表是一项繁重但并不经济的任务

Heavy UI updates as transitions​

使用 useTransition React 将紧急任务的呈现（在用户键入时更新输入字段）与非紧急任务（突出显示列表中的查询）分开。这种方法提供了更好的用户体验。

React 中的并发模式可以让将紧急任务与非紧急任务分开 使 UI 更新更加人性化
